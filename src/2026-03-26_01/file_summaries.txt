=== api.py ===
The `api.py` script automates summarizing a Python codebase using the OpenAI GPT model.

**Purpose:**  
To generate concise summaries of individual Python files and then produce a high-level overview of the entire codebase.

**Key Functions:**  
- `get_python_files(root=".")`: Recursively finds all `.py` files from the root directory.  
- `read_file(path)`: Reads and truncates file content to a max character limit (20,000 chars).  
- `call_llm(prompt, content)`: Sends prompts and content to the OpenAI GPT model (`gpt-4.1-mini`) and returns the generated text.  
- `summarize_files(files)`: Iterates over Python files, reads their content, and generates summaries using the LLM with a focused prompt.  
- `summarize_all(summaries)`: Combines all file summaries and prompts the LLM to produce a comprehensive overview of the codebase.  
- `main()`: Orchestrates the process — finds files, summarizes them, creates the final overview, and saves results to text files.

**Important Logic:**  
- Uses a map-reduce style approach: map step generates per-file summaries; reduce step synthesizes these into a global summary.  
- Limits file reading size to avoid overly large inputs.  
- Employs structured prompts to guide the LLM for consistent and informative summaries.  
- Outputs are saved into `file_summaries.txt` (individual file summaries) and `final_summary.txt` (overall codebase summary).

Overall, the script leverages GPT to assist in understanding and documenting Python projects by summarizing code files and their collective architecture.

=== botc/encounters/base.py ===
This Python file defines a base framework for managing "encounters" in a system, likely for a game or interactive narrative.

**Purpose:**
- To provide an abstract base class and registration system for encounters that have dependencies and anti-dependencies affecting their possibility and resolution status.

**Key components:**

1. **ENCOUNTER_MAP (dict):**
   - Global registry mapping encounter names to their instantiated objects.

2. **register_encounter (decorator):**
   - Registers an encounter class by instantiating it with its name, dependencies, and anti-dependencies.
   - Adds the instance to ENCOUNTER_MAP.

3. **Encounter (abstract base class):**
   - Attributes:
     - `name`: Encounter identifier.
     - `dependency_names` & `anti_dependency_names`: Lists of names for dependencies and anti-dependencies.
     - `dependencies` & `anti_dependencies`: Lists of Encounter objects (resolved later).
     - `impossible`: True if any anti-dependency has resolved, making this encounter impossible.
     - `parents_resolved`: True if all dependencies have resolved, making this encounter possible.
     - `resolved`: True if this encounter has been completed.
     - `flavor_text` and `image`: Optional descriptive text and image for the encounter.
   - Abstract async method `resolve`: To be implemented by subclasses to define encounter resolution logic.
   - Boolean and string representations reflect encounter status.

4. **update_possibilities():**
   - Iterates over all encounters in ENCOUNTER_MAP.
   - Updates each encounter's `impossible` flag if any anti-dependency is resolved.
   - Updates `parents_resolved` flag if all dependencies are resolved.

5. **reset_resolved():**
   - Resets all encounters' `impossible`, `parents_resolved`, and `resolved` flags to False.
   - Intended mainly for testing purposes.

**Important logic:**
- Encounters become impossible if any anti-dependency resolves.
- Encounters become possible only if all dependencies have resolved.
- The system uses these flags to control which encounters can occur next.
- The abstract `resolve` method is asynchronous, anticipating potentially complex or time-dependent resolution logic.

Overall, this module sets up a dependency-driven encounter system with clear rules for possibility and resolution, designed for extensibility via subclassing and registration.

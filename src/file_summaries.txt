=== botc/enums.py ===
1) Purpose of the file
- Centralizes core game enums and a mutable status container for a Blood on the Clocktower–style engine.
- Provides role taxonomy (RoleClass) and concrete roles (RoleName) with metadata for categorization and display.
- Supplies utility properties and methods for game logic (e.g., reliability under poison/drunk, protection handling, role lookup by class).

2) Key classes/functions
- Status (dataclass)
  - Fields: poisoned, drunk, protected, alive, red_herring (all bool with sensible defaults).
  - is_reliable: True when not poisoned and not drunk; used to gate trustworthy info/actions.
  - is_protected: True when protected and not poisoned; poison nullifies protection.
  - toggle(attr_name): Dynamically flips a named boolean attribute; validates existence, else raises ValueError.
- Alignment (Enum)
  - GOOD, EVIL; __str__ returns title-cased name for user-friendly display.
- RoleClass (Enum)
  - GAMEMASTER, DEMONS, MINIONS, OUTSIDERS, TOWNSFOLK.
  - display_name and __str__: title-case with underscores replaced by spaces.
- RoleName (Enum)
  - Enumerates roles and attaches a role_class to each member via custom __new__.
  - Members cover GM plus Trouble Brewing–style Minions, Outsiders, Townsfolk, and Imp (Demon).
  - display_name and __str__: human-friendly names.
  - get_by_class(target_class): Returns a List[RoleName] filtered by RoleClass.

3) Important logic or algorithms
- Enum member augmentation: RoleName.__new__ sets both the integer value and an extra attribute (role_class), enabling per-member metadata without separate mappings.
- Reliability/protection rules encoded as properties:
  - is_reliable excludes poisoned/drunk states (mirrors game mechanics where such states corrupt info).
  - is_protected is suppressed by poison, modeling poison overriding protective effects.
- Dynamic state mutation: Status.toggle uses hasattr/getattr/setattr to flip flags by string, supporting flexible UI/command handling.
- Role lookup: get_by_class uses a concise list comprehension over the enum members for fast categorization.

4) Notable dependencies or patterns
- dataclasses for a lightweight, mutable state container with defaulted fields.
- enum.Enum with:
  - Custom __new__ to embed metadata per member.
  - __str__/display_name to decouple internal identifiers from UI text.
- Typing: List[RoleName] return type for clarity; forward reference used in annotation.
- Consistent display formatting pattern (replace underscores, title-case) across enums to standardize UI/printing.
- Stable integer values assigned to RoleName and RoleClass, useful for serialization or ordering.

=== botc/player.py ===
1) Purpose of the file
- Defines the Player entity for a Blood on the Clocktower-like game. Encapsulates identity (names and roles), alignment, status flags, role-driven behavior, and action execution within the game loop.

2) Key classes/functions
- Player
  - __init__(player_name, believed_role, registered_role, registered_alignment): Initializes core player state. Derives alignment if not provided, instantiates status, binds a role behavior based on believed_role, and sets vote tokens.
  - show_role(): Returns a simple greeting string identifying the player’s believed role.
  - _default_alignment(): Infers alignment from the registered role’s RoleClass (EVIL for DEMONS/MINIONS, otherwise GOOD).
  - take_action(game): Async entry point for a player’s nightly action; skips dead players except RAVENKEEPER, then delegates to the role behavior’s act().
  - _to_dict(): Serializes selected player fields (roles, alignment, and status flags) to a dict. Note: contains a duplicate "registered_role" key.

3) Important logic or algorithms
- Alignment inference: If no registered_alignment is supplied, it is computed from registered_role.role_class, mapping demons/minions to EVIL and all others to GOOD.
- Action gating: Only alive players act, except RAVENKEEPER who may act when dead (rule-specific exception). The actual action is delegated to the bound role behavior’s async act(self, game).
- Behavior selection: role_behavior is chosen via BEHAVIOR_MAP using the believed_role (not the registered_role), enabling a separation between what the player thinks they are and what they really are (useful for roles like Drunk/Poisoned or registration effects). A PassiveBehavior is used if no mapping is found.

4) Notable dependencies or patterns
- Strategy/behavior pattern: BEHAVIOR_MAP maps RoleName to a RoleBehavior; behaviors implement an async act(). PassiveBehavior provides a no-op fallback.
- Async integration: take_action is async to fit into an event-driven game loop managed by GameManager.
- Type hinting patterns: Uses from __future__ import annotations and TYPE_CHECKING imports to avoid runtime circular dependencies (GameManager, RoleBehavior).
- Enum-driven domain model: RoleName, RoleClass, and Alignment enums are central; RoleName appears to carry metadata (role_class). Status, though imported from enums, acts like a state container with attributes alive/poisoned/protected.
- Potential issues:
  - _to_dict has a duplicated "registered_role" key; the last one wins but it’s redundant and error-prone; also player_name and vote_token are omitted.
  - show_role prints the enum object directly; may produce verbose values unless RoleName.__str__ is customized (often .name or a display string is preferable).

=== botc/discord_manager/polling.py ===
1) Purpose of the file
- Orchestrates Discord-based polling for a Blood on the Clocktower game. Provides two async workflows:
  - A GM volunteer poll using Discord’s native Poll feature, selecting and updating the game’s Game Master.
  - A day execution vote using a custom dropdown view, tallying majority results and updating game state.

2) Key classes/functions
- class PollManager
  - __init__(game: GameManager)
    - Stores a reference to the game manager for state updates.
  - async run_gamemaster_poll(interaction, allowed_player_names) -> Optional[discord.Message]
    - Creates a public thread and posts a Discord Poll (“Yes” / “No / Skip”) asking who wants to be GM.
    - Waits up to ~4 minutes (or earlier if enough votes accumulated), ends the poll, filters “Yes” voters to allowed list (by username/global_name), randomly selects a GM, and updates/removes them from relevant game lists. Returns the poll message.
  - async run_execution_poll(interaction, allowed_player_ids) -> Optional[discord.Message]
    - Creates a public thread with an embed and attaches an ExecutionView dropdown (target selection).
    - Waits up to ~4 minutes or until all allowed voters have cast a vote; disables input, tallies votes (alive players + “Skip Vote”), requires strict majority (>50%) for execution, updates the executed player’s alive status and game.executed_player, and posts the outcome. Returns the poll message.

3) Important logic or algorithms
- Channel/type and ephemeral interaction handling:
  - Both methods require a TextChannel; they use interaction.response vs followup depending on whether the response is already sent.
- GM poll:
  - Poll creation via discord.Poll (duration set to 1 hour, but waiting loop only ~4 minutes).
  - Safe polling loop: sleep 5s intervals, refetch message, catch discord.HTTPException, early-exit when total votes across answers >= number of allowed players.
  - Poll termination: attempts poll_message.end_poll(); on failure, refetches the message.
  - Candidate filtering: normalizes allowed names (strip “@”, lowercase) and matches against user.name or user.global_name (lowercased).
  - Random selection: random.choice among filtered “Yes” voters; updates game.game_master and removes that player from game.players, game.players_alive, and game.player_names if present.
- Execution poll:
  - Uses a custom ExecutionView; votes stored in view.dropdown.votes as {voter_key: chosen_target}.
  - Wait loop ends when all allowed voters are present in votes or time expires; then disables the dropdown and edits the message.
  - Tally: builds a results dict for alive players plus “Skip Vote”; counts only votes from allowed voters; determines strict majority (> total_valid_votes / 2.0).
  - Outcomes: no votes => no execution; majority “Skip Vote” => no execution; otherwise marks player dead and records executed_player.
- Edge cases/nuances:
  - GM loop’s early-exit counts all poll votes (not only from allowed players), which may end the wait prematurely if outsiders vote.
  - Poll duration (1h) vs internal wait (4m) mismatch; code ends poll explicitly afterward.
  - Execution voting key semantics: comments indicate votes keyed by “username,” but parameter is named allowed_player_ids (strings). Logic assumes these keys match; mismatch would invalidate votes.
  - Majority uses strict “> 50%” (ties or exactly half do not pass).

4) Notable dependencies or patterns
- Discord.py features:
  - discord.Poll, PollAnswer.voters(), poll_message.end_poll(), discord.Embed, discord.ui style View (ExecutionView), public threads, Interaction followup/response.
  - discord.utils.get for poll answer retrieval.
- Async patterns:
  - Periodic polling with asyncio.sleep; robust fetch with HTTP exception handling; view disabling and message edit to lock input.
- GameManager integration:
  - Reads/updates: game.day_counter, game.game_master, game.players, game.players_alive, game.player_names, game.executed_player.
  - Methods: get_player_by_name, get_board_str.
- General patterns:
  - Input normalization (strip “@”, lowercase) and name/global_name matching.
  - Defensive error handling (channel checks, HTTP exceptions, missing players, null poll state).
  - Random selection for fairness when multiple GM volunteers exist.

=== botc/discord_manager/views.py ===
1) Purpose of the file
- Defines Discord UI components (Views, Selects, Buttons) to support a Blood on the Clocktower–style game flow:
  - Joining a lobby
  - Conducting an execution vote among alive players
  - A reusable generic dropdown selector for other prompts

2) Key classes/functions
- ExecutionDropdown (discord.ui.Select)
  - Builds a single-select dropdown with all alive players (labelled by player.player_name) plus a “Skip Vote” option.
  - Records selections in a local dict votes mapping Discord username -> chosen label.
  - Sends an ephemeral confirmation on each selection; allows changing selection until the poll is closed (component not disabled).
- ExecutionView (discord.ui.View)
  - Wraps an ExecutionDropdown and exposes it as a view (timeout=None).
- JoinLobbyView (discord.ui.View)
  - Persistent button “Join Game” (custom_id="join_game_btn") that adds the interacting user’s name to game.player_names and confirms ephemerally.
  - If already joined, responds with the current player list and a warning (see caveat about interaction responses below).
- DropdownView (discord.ui.View)
  - Generic dropdown supporting up to 25 options and N-way selection (min=1, max=max_selection).
  - On selection: stores selected_values, disables the component, edits the message to acknowledge, and calls self.stop() to unblock await view.wait() in calling code.
  - Intended for single-user flows tied to a specific message.

3) Important logic or algorithms
- Dynamic option construction:
  - ExecutionDropdown converts a list of alive player objects to SelectOptions by label=player.player_name and adds a “Skip Vote” sentinel.
- Vote collection strategy:
  - ExecutionDropdown stores votes by Discord username (interaction.user.name) to the chosen option label. This enables re-voting (overwriting) but:
    - Uses display names rather than unique IDs, which can collide if two users share the same name or if users rename themselves mid-game.
    - Stores state in-memory on the dropdown instance; not persisted across restarts or message re-creations.
- Interaction flow patterns:
  - Execution voting allows updates until the view is manually closed (component remains enabled).
  - The generic DropdownView locks after the first selection by disabling the menu and stopping the view; appropriate for single-user prompts but would block other users if used in a shared message.
- Timeouts and persistence:
  - ExecutionView and JoinLobbyView use timeout=None; the button in JoinLobbyView has a custom_id, making it suitable for persistent views across restarts if registered (bot.add_view). The select in ExecutionView lacks a custom_id, so it won’t persist across restarts even if the view is re-registered.
- Interaction response caveat in JoinLobbyView:
  - In the “already joined” branch, it calls interaction.response.send_message twice; Discord interactions allow only one initial response. Subsequent messages must use interaction.followup.send. As written, this will raise if executed.

4) Notable dependencies or patterns
- discord.py UI components (discord.ui.View, discord.ui.Select, discord.ui.Button, discord.SelectOption) and Interaction API (ephemeral replies, edit_message).
- TYPE_CHECKING gate and from __future__ import annotations to avoid runtime import cycles and enable forward type references (GameManager).
- Use of custom_id for persistent button; absence of custom_id for selects implies non-persistent components.
- Direct callback assignment (self.select.callback = self.select_callback) instead of decorator-based component definitions, enabling dynamic construction.
- Ephemeral confirmations are used liberally to avoid channel spam and provide private feedback.

=== botc/encounters/base.py ===
1) Purpose of the file
- Defines the core “Encounter” abstraction and lifecycle flags for a simple dependency-driven encounter system.
- Provides a registry via a class decorator to auto-instantiate and register encounters by name, with declared dependency and anti-dependency names.
- Offers utilities to recompute encounter availability based on resolved dependencies and to reset state (useful for tests).

2) Key classes/functions
- ENCOUNTER_MAP: Global dict[str, Encounter], mapping encounter names to their single instantiated Encounter objects.
- register_encounter(encounter_name, dependency_names, anti_dependency_names): Class decorator that instantiates the decorated class with those three parameters and registers the instance in ENCOUNTER_MAP. Returns the class (not the instance).
- class Encounter(ABC):
  - __init__(name, dependency_names, anti_dependency_names): Stores name and string lists of dependency names; initializes reference lists (dependencies, anti_dependencies), state flags (impossible, parents_resolved, resolved), and optional presentation fields (flavor_text, image).
  - async resolve(...): Abstract; subclasses implement how the encounter resolves (likely sets resolved=True and side effects).
  - __str__(): Returns a concise debug string with name, parents_resolved, impossible.
  - __bool__(): True if parents_resolved is True and impossible is False (enables “if encounter:” checks to mean “encounter is possible”).
- update_possibilities(): For every registered encounter, recomputes:
  - impossible: currently set True only if all anti_dependencies are resolved; False if any anti_dependency is not resolved or there are none.
  - parents_resolved: True only if all dependencies are resolved (vacuously True if no dependencies).
- reset_resolved(): Sets impossible=False, parents_resolved=False, resolved=False for all encounters (intended mainly for testing; call update_possibilities afterward to recompute availability).

3) Important logic or algorithms
- Availability model:
  - parents_resolved = all(dep.resolved for dep in dependencies) (empty list => True).
  - impossible is computed as all(anti.resolved for anti in anti_dependencies) or False if there are none.
  - Encounter truthiness (__bool__) is parents_resolved and not impossible.
- Notable inconsistency/bug: The class comment states “if any of these anti-dependencies resolve, this encounter becomes impossible,” but update_possibilities marks impossible=True only when all anti-dependencies have resolved. The intended logic likely should be impossible = any(anti.resolved for anti in anti_dependencies).
- Dependencies vs names: dependency_names/anti_dependency_names are stored, but this module does not wire them into dependencies/anti_dependencies object references; an external step must populate those lists.

4) Notable dependencies or patterns
- Pattern: Decorator-based registry with eager instantiation (one instance per decorated class, singleton-like via ENCOUNTER_MAP).
- Typing: from __future__ import annotations enables forward references; lists are annotated as List[Encounter].
- Async design: resolve is async, implying integration with an event loop; update/reset are synchronous and global.
- PIL: Optional Image.Image for encounter visuals.
- Minor issues: random is imported but unused; a comment typo (“impossible always takes precedence over impossible”).

=== botc/encounters/deck.py ===
1) Purpose of the file
- Defines a deck system for “encounters” in a game. It loads a decklist from packaged JSON, represents encounters as draw-able cards (with rarity via duplicate cards), enforces prerequisite/possibility constraints when drawing, and tracks resolved vs unresolved cards.

2) Key classes/functions
- EncounterCard
  - Thin wrapper holding a specific_encounter (Encounter). Multiple cards can reference the same Encounter to weight draw probability.
- Deck
  - from_json(json_file_name): Classmethod that loads a decklist from botc.decklists/<name>.json. Expects keys "encounter_name_list" and "count_list". Resolves names via ENCOUNTER_MAP and returns a Deck instance.
  - __init__(encounter_list, count_list): Builds the unresolved deck by creating count copies of EncounterCard per Encounter; initializes constraints via update_possibilities().
  - draw_card() -> EncounterCard: Filters unresolved cards to those whose encounter.parents_resolved is True and encounter.impossible is False; randomly selects one, marks its encounter as resolved, moves the card to resolved, calls update_possibilities(), and returns the card.
  - __str__(): Summarizes unresolved/resolved cards by encounter name with frequencies.
  - __len__(): Returns total number of cards (resolved + unresolved).
  - __bool__(): Returns False if no unresolved cards; otherwise True if any unresolved card has a non-null specific_encounter (effectively True if unresolved is non-empty).

3) Important logic or algorithms
- Eligibility filtering on draw: Only encounters whose prerequisites are satisfied (parents_resolved) and not marked impossible can be drawn. Selection is uniform among eligible cards; duplicate cards weight probabilities linearly.
- State propagation: Upon draw, encounter.resolved is set to True and update_possibilities() is called to recompute global feasibility (e.g., unlocking dependents or marking others impossible). Because multiple cards can reference the same Encounter instance, resolving one affects all of its duplicate cards.
- Data-driven deck construction: Deck composition is externalized in JSON, mapping encounter names to counts; ENCOUNTER_MAP translates names to Encounter instances.
- Edge cases/nuances:
  - If no cards are eligible (all fail the filter) but unresolved cards remain, random.choice([]) will raise an exception; __bool__ still returns True in this situation.
  - __len__ counts resolved cards too; not a measure of remaining drawables.
  - zip(encounter_list, count_list) silently truncates to the shorter list if lengths mismatch.
  - Deck mutates Encounter objects from ENCOUNTER_MAP (singletons), so multiple Decks/games in the same process would share state.

4) Notable dependencies or patterns
- Depends on botc.encounters.base:
  - Encounter: expected to expose name, resolved, impossible, parents_resolved.
  - update_possibilities(): side-effect function to recalculate encounter feasibility.
  - ENCOUNTER_MAP: name -> Encounter singleton mapping.
- Uses importlib.resources.files to load packaged JSON resources from botc.decklists.
- Randomness via random.choice; no seeding or reproducibility controls exposed.
- Pattern: duplicates-as-weighting (multiple cards per Encounter) instead of adjusting probabilities directly.
- Implements Python dunders for integration with idioms (truthiness, length, string representation).
- Type hints used for clarity; minimal object model (EncounterCard could be a dataclass but isn’t).

=== botc/encounters/__init__.py ===
1) Purpose of the file
- Package initializer that wires up encounter/event objects after all are instantiated.
- Resolves forward references: converts dependency names (strings) into actual object references on each encounter.
- Triggers a finalization step (update_possibilities) once dependencies are wired.
- Exposes Deck at the package level for convenient import (from botc.encounters import Deck).

2) Key classes/functions
- ENCOUNTER_MAP (from .base): Central registry mapping encounter names -> encounter instances. Each instance is expected to have:
  - dependency_names: list[str] of forward-referenced dependencies
  - anti_dependency_names: list[str] of forward-referenced anti-dependencies
  - dependencies: list[Encounter] to be populated with actual instances
  - anti_dependencies: list[Encounter] to be populated with actual instances
- update_possibilities (from .base): Recomputes derived state (e.g., which encounters are eligible/possible) after dependencies are resolved.
- Deck (from .deck): Public class representing an encounters deck (details defined in deck module).
- Submodules imported: base, simple, deck — their import side effects are relied upon to register/populate ENCOUNTER_MAP.

3) Important logic or algorithms
- Two-phase initialization (forward-reference resolution):
  - Phase 1 (in submodules): Encounter objects are created and registered in ENCOUNTER_MAP with only names for dependencies/anti-dependencies.
  - Phase 2 (here): For each encounter in ENCOUNTER_MAP, map each name in dependency_names and anti_dependency_names to the corresponding object in ENCOUNTER_MAP and append it to dependencies/anti_dependencies.
- Finalization: Calls update_possibilities() after all links are established so any logic depending on the dependency graph is consistent.
- Complexity: Linear in the number of dependency edges (O(N + M); N encounters, M total dependency + anti-dependency links).
- Executed at import time; assumes ENCOUNTER_MAP is fully populated by importing base and simple first.

4) Notable dependencies or patterns
- Delayed wiring/forward-reference pattern to avoid circular construction issues across modules.
- Module-level side effects: Importing the package performs graph wiring and state finalization once (per interpreter session, due to module caching).
- Import order dependencies: base and simple must register encounters before resolution (ensured by from . import base,deck,simple at the top).
- Potential caveats:
  - Lists are appended to without clearing; reloading the module could duplicate links.
  - Missing names in ENCOUNTER_MAP would raise KeyError (no validation/guarding here).
- Public API: Deck is re-exported, making botc.encounters a convenience entry point for consumers.

=== botc/discord_manager/cogs/game_commands.py ===
1) Purpose of the file
- Defines a Discord Cog (GameCommands) that orchestrates core game-management flows for a Blood on the Clocktower bot:
  - Lobby management (open lobby, list players)
  - Seating order configuration via mentions
  - Kicking off game start and GM selection via a poll
  - Sending private UI prompts to players via DMs (dropdowns and polls)
  - Utility command to print guild users to console

2) Key classes/functions (with brief descriptions)
- GameCommands(commands.Cog): Cog binding bot and game services
  - __init__(bot, game): Stores BotManager and GameManager references
  - set_seats(interaction, mentions): Admin slash command to lock seating order based on mentioned users. Validates roster and reorders self.game.players/self.game.player_names; shows board via game.get_board_str()
  - dmdropdown(user_name, message_text, options, max_selection) -> Optional[List[str]]:
    - Sends a Discord UI dropdown (custom DropdownView) to a user’s DMs; waits for selection or timeout; returns chosen values
  - dmpoll(user_name, poll_message, poll_options, max_selection) -> Optional[List[str]]:
    - Sends a Discord Poll in DMs (if ≤10 options and multiple choice allowed); otherwise falls back to dmdropdown
    - Polls for results by refetching the message periodically; enforces max_selection and retries if exceeded
  - send_direct_message(user_name, message) -> bool: DMs a user by name/global_name; returns success flag; logs errors
  - print_users(ctx): Prefix command (admin) that prints guild member info to terminal; acknowledges in channel
  - open_lobby(interaction): Admin slash command that resets lobby state and posts a JoinLobbyView for players to join
  - start_game(interaction): Admin slash command that closes lobby, runs GM poll via PollManager, and starts the game via GameManager.start_game()
  - display_players(interaction): Lists current players from self.game.players

3) Important logic or algorithms
- Seating order via mentions
  - Extracts mentioned user IDs with regex r'<@!?(\d+)>'
  - Validates the number of mentions matches current game players
  - Resolves users (cache first, then fetch) and maps to user.name to derive new order
  - Verifies the new order matches current roster by name, then reconstructs self.game.players accordingly
  - Potential pitfalls:
    - Roster matching compares case-insensitive current_names to case-sensitive new_order_names; may misreport mismatches
    - Compares by user.name only (not IDs), which can be non-unique or change
- DM dropdown flow
  - Enforces 25-option limit (Discord UI constraint)
  - Looks up user by exact match of name/global_name (lowercased) in bot.users; no fetch fallback here
  - Waits on view; on timeout, disables the select and edits message; returns None
- DM poll flow
  - Uses discord.Poll (duration 1 hour) with multiple based on max_selection > 1
  - Refresh loop fetches the message every 2 seconds to detect any votes; for multi-select, adds a 3-second buffer before finalizing
  - If selected_count > max_selection, ends poll, informs the user, and restarts the poll recursively
  - Gathers chosen answers by vote_count > 0 and ends the poll
  - Falls back to dropdown if >10 options or single-select requested
- Game start and lobby
  - open_lobby resets game.player_names and posts JoinLobbyView
  - start_game runs PollManager.run_gamemaster_poll over game.player_names, then GameManager.start_game()
  - Notable anomalies:
    - Debug code injects many Player instances with placeholder name "iiiii5184" for every townsfolk role
    - Minimum players check uses num_players < 0 (likely intended < 5), so it never blocks
- Utility/user lookup patterns and error handling
  - Name/global_name lookups rely on bot.users cache; set_seats has a fallback fetch_user by ID, but DM helpers don’t
  - Handles discord.Forbidden and discord.HTTPException on DMs; logs to console
  - Uses ephemeral responses for most admin interactions; however, print_users uses ephemeral=True on ctx.send (not supported for text commands)

4) Notable dependencies or patterns
- Discord API usage:
  - discord.ext.commands Cog pattern with both slash (app_commands) and legacy prefix commands
  - Discord components via custom views (JoinLobbyView, DropdownView); view.wait pattern and component disabling on timeout
  - Discord Polls API (discord.Poll, add_answer, end_poll) used in DMs with periodic refetch to observe state
- Internal dependencies:
  - GameManager for game state, get_board_str(), start_game()
  - PollManager for GM selection workflow
  - RoleName/RoleClass for role categorization (used in debug player injection)
- Patterns and caveats:
  - Heavy reliance on username/global_name string matching; IDs would be safer
  - Redundant imports (Player, RoleName, Alignment repeated)
  - Mixed command paradigms (slash vs prefix) with an ephemeral misuse on a prefix command
  - Asynchronous polling loop with timeouts ensures responsiveness but adds API churn (fetch_message every 2 seconds)

=== botc/enums.py ===
1) Purpose of the file
- Centralizes game-related enumerations and a status model for a Blood on the Clocktower–style engine.
- Provides human-friendly display helpers and utilities to group/filter roles by class.
- Encapsulates player condition effects (poisoned, drunk, protected, etc.) with derived logic for reliability and protection.

2) Key classes/functions
- Status (dataclass)
  - Fields: poisoned, drunk, protected, alive, red_herring (all bools; defaults set).
  - is_reliable (property): True iff not poisoned and not drunk.
  - is_protected (property): True iff protected and not poisoned (poison negates protection).
  - toggle(attr_name): Dynamically toggles a named flag; raises ValueError if unknown.
- Alignment (Enum)
  - Members: GOOD, EVIL.
  - __str__: Title-cased name for display.
- RoleClass (Enum)
  - Members: GAMEMASTER, DEMONS, MINIONS, OUTSIDERS, TOWNSFOLK.
  - display_name: Human-readable name (underscores to spaces, title-cased).
  - __str__: Returns display_name.
- RoleName (Enum)
  - Extends Enum members with extra attribute role_class via custom __new__.
  - Members: GAMEMASTER; Minions (POISONER, SPY, BARON, SCARLET_WOMAN); Outsiders (BUTLER, DRUNK, RECLUSE, SAINT); Townsfolk (WASHERWOMAN, LIBRARIAN, INVESTIGATOR, CHEF, EMPATH, FORTUNE_TELLER, UNDERTAKER, MONK, RAVENKEEPER, VIRGIN, SLAYER, SOLDIER, MAYOR); Demons (IMP).
  - display_name and __str__: Same formatting as RoleClass.
  - get_by_class(target_class): Returns list of RoleName members matching a given RoleClass.

3) Important logic or algorithms
- Derived status semantics:
  - Reliability is strictly invalidated by poisoned or drunk.
  - Protection is invalidated by poisoned (even if protected is True).
- Enum with payload pattern:
  - RoleName.__new__ stores an int value and attaches role_class metadata per member, enabling quick grouping/filtering of roles.
- Consistent display transformation:
  - __str__ and display_name unify how enums render to user-friendly strings.
- Role grouping:
  - RoleName.get_by_class provides a simple query mechanism for role classification.

4) Notable dependencies or patterns
- dataclasses.dataclass for a lightweight, mutable state container (Status).
- enum.Enum with custom __new__ to add per-member attributes (RoleName → role_class).
- Type hints (typing.List, RoleClass annotations) for clarity and tooling support.
- Defensive API design in Status.toggle with attribute existence check and explicit error on invalid keys.
- Separation of concerns:
  - Alignment is defined but decoupled from RoleName/RoleClass (no direct mapping enforced).
  - Role classifications and IDs are explicit, making it easy to extend by adding new Enum members with (value, RoleClass).

=== botc/player.py ===
1) Purpose of the file
- Defines the Player model for a Blood on the Clocktower-like game. It encapsulates identity, perceived vs actual role/alignment, current status, voting token, and delegates in-game actions to a role-specific behavior.

2) Key classes/functions
- class Player
  - __init__(player_name, believed_role, registered_role, registered_alignment): Initializes player identity, resolves alignment (falls back to a default derived from the registered role if not provided), creates a Status, selects a RoleBehavior from BEHAVIOR_MAP based on believed_role (default PassiveBehavior), and sets a vote_token.
  - show_role(): Returns a user-facing string informing the player of their believed_role.
  - _default_alignment(): Returns Alignment.EVIL if the registered_role.role_class is DEMONS or MINIONS; otherwise Alignment.GOOD.
  - take_action(game): Async. If the player is dead and does not believe they are the Ravenkeeper, do nothing. Otherwise, await the role_behavior’s act(self, game).
  - _to_dict(): Serializes selected fields to a dict using enum .name values and status booleans.

3) Important logic or algorithms
- Behavior resolution via Strategy pattern: role_behavior = BEHAVIOR_MAP.get(believed_role, PassiveBehavior()) so actions are driven by the role the player believes they have.
- Dead-action gating: Disallows actions for dead players except those believing they are Ravenkeeper (special-case night action upon death).
- Alignment inference: If no registered_alignment provided, derive it from the registered_role’s RoleClass (demons/minions → EVIL; others → GOOD).
- Serialization: Produces a simple dict of names/flags for external use.
- Notable caveats:
  - _to_dict duplicates the "registered_role" key (second occurrence overwrites the first; functionally harmless but redundant).
  - registered_alignment is typed as Alignment but treated as optional in __init__ (should likely be Optional[Alignment] with a default of None).
  - role_behavior is fixed at init; if believed_role changes later, behavior won’t auto-update.
  - Status is instantiated from botc.enums, which is atypical naming for a non-enum class (assumes a class with alive/poisoned/protected attributes).

4) Notable dependencies or patterns
- botc.enums: Alignment, RoleClass, RoleName, Status (enums plus a status holder).
- botc.behaviors: BEHAVIOR_MAP (role → behavior mapping) and PassiveBehavior (no-op default).
- Async integration: take_action and RoleBehavior.act are coroutines to fit an asynchronous GameManager loop.
- Type-hint hygiene and circular import avoidance: from __future__ import annotations and TYPE_CHECKING imports (GameManager, RoleBehavior) used only for hints.

=== botc/discord_manager/polling.py ===
1) Purpose of the file
- Orchestrates Discord-based polling for a Blood on the Clocktower game. It runs:
  - A Game Master volunteer poll (using Discord’s native Polls) and randomly selects a GM from eligible volunteers.
  - A daily execution vote (custom dropdown UI) and updates game state based on majority results.

2) Key classes/functions
- PollManager(game: GameManager)
  - Holds a reference to the game state (GameManager) and coordinates polling flows.

- async run_gamemaster_poll(interaction, allowed_player_names) -> Optional[discord.Message]
  - Validates the command is used in a text channel and sends ephemeral status.
  - Creates a public thread and posts a Discord Poll (“Do you want to be a GM?” with Yes/No).
  - Periodically refetches the poll message, ending early if the total number of votes reaches the number of allowed players or after a timeout (4 minutes).
  - Ends the poll, collects all “Yes” voters, filters them to allowed players (by normalized username/global_name), randomly selects one as GM, and updates GameManager:
    - Sets game_master to the selected username.
    - Attempts to remove the selected GM from game.players and game.players_alive (if present).
    - Removes their name from game.player_names (string list).
  - Posts the outcome in the thread. Returns the poll message.

- async run_execution_poll(interaction, allowed_player_ids) -> Optional[discord.Message]
  - Validates text channel, sends ephemeral status, creates a public thread.
  - Sends an embed with an ExecutionView dropdown (choices: alive players + “Skip Vote”).
  - Waits up to 4 minutes, or until all allowed players have voted (tracked via view.dropdown.votes).
  - Disables the dropdown, edits the message, and tallies votes:
    - Counts only votes from allowed_player_ids.
    - Results keyed by player_name and “Skip Vote”.
    - Determines execution only if a target has > 50% of valid votes cast.
  - If executed, sets the player’s alive status to False and records executed_player; otherwise, announces no execution.
  - Prints the board state for debugging and returns the message.

3) Important logic or algorithms
- Interaction response handling: Uses interaction.response vs interaction.followup based on is_done() to avoid double-response errors.
- Thread isolation: Each poll occurs in a dedicated public thread to reduce channel clutter.
- GM poll monitoring:
  - Loops every 5s to refetch the poll message; handles discord.HTTPException safely.
  - Early exit when the total (Yes+No) votes meet the number of allowed players or when timeout hits.
  - Ends the poll programmatically and refetches on error if needed.
- Candidate filtering for GM:
  - Normalizes allowed names (strip leading @, lowercase).
  - Compares against user.name and user.global_name (lowercased) to qualify volunteers.
  - Random selection among qualified volunteers.
- Execution vote tally:
  - Counts only votes from allowed voters.
  - Majority threshold is strictly > 50% of valid votes cast (not of living players).
  - “Skip Vote” acts as a non-execution option if it wins a majority.
  - Disables UI after completion to freeze results.

4) Notable dependencies or patterns
- Discord.py features:
  - discord.Interaction, discord.TextChannel, discord.Thread (public_thread), discord.Embed.
  - discord.Poll, PollAnswer.voters(), and Message.end_poll() (native Poll API).
  - Custom UI: ExecutionView (from botc.discord_manager.views) with a dropdown that exposes votes.
- Async patterns: asyncio.sleep polling loops for timeouts and message refetching.
- GameManager integration (implied API):
  - Attributes: day_counter, players, players_alive, player_names, executed_player, game_master.
  - Methods: get_player_by_name(name), get_board_str().
- Data handling:
  - Uses usernames/global_name (string matching) for GM eligibility; uses player_name strings for execution targets.
  - allowed_player_ids expected to match keys in view.dropdown.votes (likely user IDs as strings).
- Error handling:
  - HTTPException-safe message fetching loop.
  - Graceful handling if player lookup fails during state updates.
- Misc:
  - Poll UI duration (set to 1 hour) differs from internal wait (4 minutes); logic ends the poll early.
  - Early-termination condition in GM poll considers all votes (including non-allowed), while selection filters to allowed voters.

=== botc/encounters/base.py ===
1) Purpose of the file
- Defines a base framework for “encounters” that can be resolved asynchronously and are gated by dependencies and anti-dependencies.
- Provides a global registry and utilities to compute which encounters are currently possible/impossible based on other encounters’ resolution state.

2) Key classes/functions
- ENCOUNTER_MAP: Global dict[str, Encounter] storing a single instantiated Encounter per registered name.
- register_encounter(encounter_name, dependency_names, anti_dependency_names):
  - Class decorator that eagerly instantiates the decorated class with the given name and dependency name lists, and registers the instance in ENCOUNTER_MAP.
  - Returns the class unchanged; registration happens as a side effect at import time.
- class Encounter(ABC):
  - __init__(name, dependency_names, anti_dependency_names):
    - Stores name and dependency name lists.
    - Initializes dependency object lists (dependencies, anti_dependencies), and state flags: impossible, parents_resolved, resolved.
    - Optional metadata: flavor_text, image (PIL Image).
  - async resolve(...): Abstract; subclasses implement encounter resolution and should set self.resolved = True when done.
  - __str__(): Debug-friendly string with name, parents_resolved, impossible.
  - __bool__(): Encounter is truthy only when parents_resolved is True and impossible is False (impossible takes precedence over possible).
- update_possibilities():
  - Recomputes, for each registered encounter:
    - impossible: based on anti_dependencies’ resolved states (see logic below).
    - parents_resolved: True only if all dependencies are resolved (or there are none).
- reset_resolved():
  - Resets impossible, parents_resolved, resolved to False for all registered encounters (useful for tests).

3) Important logic or algorithms
- Possibility semantics:
  - parents_resolved is True iff all dependencies are resolved (logical AND across dependencies). If no dependencies, it becomes True.
  - impossible is computed from anti_dependencies:
    - Current code: impossible becomes True only if there is at least one anti-dependency and all anti-dependencies are resolved; it is False if any anti-dependency is not resolved, or if there are none.
    - Comment intent vs. implementation mismatch: The comment says “if any anti-dependency resolves, this encounter becomes impossible,” which would be logical OR; the implementation uses logical AND (all must be resolved) to mark impossible. This is likely a bug or design discrepancy.
- Truthiness:
  - __bool__ enforces “possible” as parents_resolved AND NOT impossible, giving impossible precedence.
- Registry lifecycle:
  - The decorator creates a singleton instance per encounter at import time. Dependency wiring by names to actual Encounter instances must occur elsewhere (not provided here). Until wired, dependencies/anti_dependencies remain empty, which makes encounters appear possible (parents_resolved=True, impossible=False) after update_possibilities.

4) Notable dependencies or patterns
- Patterns:
  - Global registry of singleton-like encounter instances keyed by name.
  - Decorator-based eager registration/instantiation.
  - Name-based dependency injection: stores dependency_names/anti_dependency_names alongside object lists; expects an external linking step.
  - Overwrites silently on duplicate names in ENCOUNTER_MAP (no guard).
  - State recomputation is O(N * (deps + anti_deps)) across all encounters; no incremental or topo-sorted update.
- Libraries/typing:
  - abc.ABC and abstractmethod for extensibility.
  - Asynchronous resolve method (async/await friendly).
  - PIL.Image for optional image metadata (type only in this module).
  - __future__.annotations enables forward references in type hints.
  - random is imported but unused in this file (likely for subclasses).

=== botc/encounters/deck.py ===
1) Purpose of the file
- Defines a deck system for “encounters” in a game. It loads a decklist from packaged JSON, represents encounter weighting via multiple “cards” pointing to the same Encounter, enforces dependency gating when drawing (via Encounter fields and a global update_possibilities), and tracks resolved vs unresolved cards.

2) Key classes/functions
- EncounterCard
  - Lightweight wrapper around a specific Encounter instance. Multiple cards can reference the same Encounter to model rarity/weighting.
- Deck.from_json(json_file_name)
  - Factory that reads a decklist JSON from the botc.decklists package (ensuring a .json extension).
  - Expects keys: "count_list" (List[int]) and "encounter_name_list" (List[str]).
  - Maps encounter names to Encounter instances via ENCOUNTER_MAP and constructs a Deck.
- Deck.__init__(encounter_list, count_list)
  - Builds the unresolved card list by replicating EncounterCard entries per count; initializes resolved list empty.
  - Calls update_possibilities() to set initial eligibility/impossibility based on current Encounter states.
- Deck.draw_card()
  - Filters unresolved cards to those whose Encounter has parents_resolved == True and impossible == False.
  - Randomly selects one, removes it from unresolved, marks its Encounter.resolved = True, appends to resolved list, calls update_possibilities(), and returns the card.
- Deck.__str__()
  - Summarizes the unresolved and resolved lists by Encounter name frequency.
- Deck.__len__()
  - Returns total number of cards across both unresolved and resolved lists (i.e., original deck size, not remaining drawables).
- Deck.__bool__()
  - True if there is at least one unresolved card (does not check if it’s drawable).

3) Important logic or algorithms
- Weighted selection by duplication
  - Rarity is modeled by duplicating EncounterCard entries for the same Encounter; selection among eligible cards is uniform, making duplicates increase draw probability.
  - All duplicated cards reference the same Encounter object (from ENCOUNTER_MAP), so marking one as resolved affects all duplicates (their Encounter state changes simultaneously).
- Gating via Encounter state
  - Eligibility to draw is controlled by Encounter.parents_resolved and Encounter.impossible (and indirectly by Encounter.resolved through update_possibilities()).
  - update_possibilities() is called after deck construction and after each draw to recompute which encounters are possible given dependency/resolution changes.
- Potential edge cases
  - draw_card() will fail (random.choice on empty list) if no unresolved cards are currently eligible (e.g., all remaining are impossible), even though __bool__ may still return True.
  - __len__ reports total deck size rather than remaining-eligible or remaining-unresolved size, which could be misleading.
  - __bool__ does not reflect eligibility; it is effectively bool(unresolved_encounter_card_list).

4) Notable dependencies or patterns
- External game state and singletons
  - Relies on ENCOUNTER_MAP (name -> Encounter) and a global update_possibilities() that mutates Encounter.impossible/parents_resolved based on current resolutions. This implies Encounter instances are shared singletons across the deck.
- Packaged resource loading
  - Uses importlib.resources.files("botc.decklists").joinpath(...).open("r") to load JSON decklists bundled with the package (Python 3.9+ style).
- JSON decklist schema
  - Assumes two parallel arrays: encounter_name_list and count_list. No explicit length/consistency validation in this file.
- Type hints and simple data wrappers
  - Employs typing.List annotations; EncounterCard is a minimal wrapper rather than a dataclass.

=== botc/encounters/__init__.py ===
1) Purpose of the file
- Centralizes and finalizes initialization of the encounters subsystem.
- Ensures all encounter objects are registered, then resolves their dependency graphs by converting string-based references to actual object references.
- Computes derived state (via update_possibilities) after dependencies are linked.
- Re-exports Deck for external use (botc.encounters.Deck).

2) Key classes/functions (as used/exported by this module)
- ENCOUNTER_MAP (from .base): Global registry mapping encounter names (str) to encounter instances. Each instance is expected to expose:
  - dependency_names: list[str] of encounter names it depends on.
  - anti_dependency_names: list[str] of encounter names it should not co-occur with.
  - dependencies: list[Encounter] to be populated with actual objects.
  - anti_dependencies: list[Encounter] to be populated with actual objects.
- update_possibilities (from .base): Post-linking routine that likely computes which encounters are valid/eligible given the resolved dependencies/anti-dependencies.
- Deck (from .deck): Exposed after setup; likely uses ENCOUNTER_MAP to construct playable/available encounter decks.

3) Important logic or algorithms
- Two-phase dependency resolution:
  - Phase 1 (elsewhere): Encounters are instantiated and registered in ENCOUNTER_MAP with only string-based dependencies (to avoid forward-reference/circular import issues).
  - Phase 2 (here): Iterate ENCOUNTER_MAP and, for each encounter:
    - For every dep_name in dependency_names, append ENCOUNTER_MAP[dep_name] to dependencies.
    - For every antidep_name in anti_dependency_names, append ENCOUNTER_MAP[antidep_name] to anti_dependencies.
  - Complexity: O(N + E), where N is number of encounters and E is total number of dependency edges.
- Derived-state update: Immediately invoke update_possibilities() after linking to compute any availability/constraint-driven state.

4) Notable dependencies or patterns
- Registry pattern: Global ENCOUNTER_MAP serves as a central source of truth for all encounters and enables cross-linking by name.
- Import side effects / plugin discovery:
  - from . import base, deck, simple ensures those modules import and (likely) register encounters into ENCOUNTER_MAP as a side effect (notably simple).
- Two-phase construction to handle forward references and avoid circular dependencies.
- Import-order sensitivity:
  - deck is imported before dependencies are resolved, but Deck is only re-exported after resolution. If deck executes logic at import time that assumes resolved dependencies, this could be fragile.
- Global mutable state:
  - dependencies and anti_dependencies are mutated at import time; this is not thread-safe during module import and assumes a single initialization pass.
  - No guard against missing keys; bad names in dependency_names/anti_dependency_names will raise KeyError during import.
  - No deduplication in dependency lists.
- Commenting style:
  - Uses a triple-quoted string as a comment explaining the two-phase initialization approach.
  - The trailing "ERROR READING FILE..." lines appear to be an artifact; if present in the file, they would cause a syntax error.
